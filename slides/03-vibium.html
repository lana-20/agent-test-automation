Architecture payoff

The delta

goto → go  ·  get_by_role() → find({})  ·  capture.screenshot instead of screenshot. That's the entire delta between the two implementations.

tools.py · Vibium dispatch()

Same return strings.
Different API calls.

tools.py — Vibium
from vibium import browser as vibium_browser

def dispatch(name: str, inputs: dict, page) -> str:
    if name == "navigate":
        page.go(inputs["url"])              # go() not goto()
        return f"Navigated to {inputs['url']}"

    if name == "click":
        el = page.find({                       # find() returns element
            "role": inputs["role"],
            "text": inputs["text"]
        })
        el.click()
        return f"Clicked {inputs['role']} '{inputs['text']}'"

    if name == "assert_visible":
        el = page.find({"role": inputs["role"], "text": inputs["text"]})
        if el and el.is_visible():
            return f"VISIBLE: {inputs['role']} '{inputs['text']}'"
        return f"FAIL: {inputs['role']} not visible"

    if name == "screenshot":
        page.capture.screenshot(path=inputs["path"])  # capture namespace
        return f"Screenshot saved to {inputs['path']}"

What happens internally

The Anthropic API spawns the Vibium MCP server, handles the tool-use loop, and returns when the task is complete. Zero orchestration code in your script.

Vibium · Critical Behavior

Always pass role
to find().

❌ Text only — unreliable

page.find({ "text": "Walk the dog" })
# Returns the OUTERMOST element containing
# that text — could be <ul> or <section>
# Clicking it does nothing useful
The outermost container element — not what you wanted to click.

✓ Role + text — correct

page.find({
  "role": "listitem",
  "text": "Walk the dog"
})
# Returns the specific <li> element
The exact <li> list item — correct and clickable.
System prompt reinforcement

The system prompt says "prefer locating by ARIA role + visible text." With Vibium's find(), this isn't just good practice — it's required for reliable targeting.

Decision Guide

When to choose
Vibium.

  • Agent is the primary browser consumer — not also writing traditional tests. Vibium's find({role, text}) is more expressive than getByRole(role, {name: text}) for AI call sites.
  • MCP integration needed — connect Claude Desktop or another AI interface to the browser. Vibium is the fastest path: one config, 85 tools available.
  • Rich capture needs — screenshots, PDFs, video recordings. Vibium's page.capture.* namespace is well-organized.
  • Chrome-first workflow — Vibium covers Chromium comprehensively. For Firefox cross-browser, see Module 04 (Selenium BiDi).
  • Rapid prototyping — MCP approach means zero wrapper code, fastest path from idea to running agent.
Module 03 · Key Takeaways

Five things
to carry forward.

1
The loop doesn't change when you change drivers
Only dispatch() changes. The model sees identical tool names and result strings.
2
Vibium API delta is small
go(), find({role, text}), capture.screenshot(). Ten minutes with the docs and you know it.
3
MCP removes the wrapper layer entirely
For prototyping and Claude Desktop use cases, one API call handles the entire loop.
4
Always pass role to find()
Text-only find() returns the outermost container — almost never what you want for interactive elements.
5
Return strings must stay consistent across modules
The model builds pattern-matching intuitions from tool result strings. Keep the phrasing identical across driver implementations.
eval checkpoint

Eval Your Vibium Agent

Vibium-specific traps to catch in eval:

1
find() selector quality — Trajectory eval should flag any find({text})-only call on an interactive element. The correct form is find({role, text}).
2
MCP consistency — Run the same task twice via MCP and via programmatic API. Finding text should match ≥ 80% of keywords in both modes.
3
Dialog handling — For pages with alerts, verify dialog_accept / dialog_dismiss appear in the tool call sequence before any post-dialog find.
Vibium's MCP mode skips the agent loop — eval it separately. MCP finding = single tool call output; programmatic = multi-turn reasoning.
