Motivation

There Is a Gap Between
"It Runs" and "It's Good."

Your agent runs without errors. It produces findings. But:

  • Did it find the right bugs?
  • Are the findings specific enough to act on?
  • Did it take sensible steps — or hallucinate selectors?
  • Will it still work after you change the model or prompt?
without evals

You discover failures in production. A model upgrade silently breaks finding quality. A prompt change halves keyword coverage. You have no baseline.

with evals

Every change is measured against a baseline. Regressions fail the CI build before they reach production. Quality improves systematically.

Architecture

Three Judges.
One Regression Harness.

1
Deterministic
Keyword matching against known expected findings. Fast, cheap, binary pass/fail. Measures correctness.
2
LLM Judge
Claude Haiku scores the finding on accuracy, specificity, and actionability. Measures quality.
3
Trajectory
Validates tool call sequence. Catches hallucinated tools and redundant loops. Measures behavior.
Regression Harness

Runs all three judges, compares aggregate scores against a committed baseline.json, exits 1 if any metric drops below threshold. Blocks the PR before it merges.

Data

Each scenario defines:

  • id — unique scenario key
  • site — URL to test
  • task — plain English instructions

Judge 1

Deterministic Judge:
Keyword Coverage

judge_deterministic.py
def judge(scenario_id, finding,
          expected_keywords, threshold=0.6):
    lower = finding.lower()
    matched = [kw for kw in expected_keywords
               if kw.lower() in lower]
    score = len(matched) / len(expected_keywords)
    return DeterministicResult(
        score=score,
        matched=matched,
        missed=[kw for kw in expected_keywords
                if kw.lower() not in lower],
        passed=score >= threshold
    )

What it measures:

  • Fraction of expected keywords present in the agent's final finding paragraph
  • Pass threshold: 60% by default — tunable per scenario severity
  • The missed list tells you exactly which concepts the agent failed to report
Example output
score=0.8
matched=['empty', 'invalid', 'username', 'error']
missed=['password']
passed=True
Judge 2

LLM Judge:
Finding Quality Rubric

Three dimensions, 0.0–1.0 each:

Accuracy
Is the finding factually correct? Does it describe what actually happened on the page?
Specificity
Does it name the exact element, selector, or behavior — not just "there was an error"?
Actionability
Would a developer know exactly what to fix from this finding alone?
overall = (accuracy + specificity + actionability) / 3  ·  pass ≥ 0.65
judge_llm.py — rubric
RUBRIC = """Score this finding on three criteria.

Finding: {finding}
Task: {task}

Score 0.0–1.0 each:
- accuracy: factually correct?
- specificity: names exact element?
- actionability: developer knows the fix?

Return JSON:
{"accuracy": float,
 "specificity": float,
 "actionability": float,
 "overall": float,
 "reasoning": "one sentence"}
"""

Judge 3

Trajectory Judge:
Did It Behave?

Keyword coverage and quality scores can both be high — but the agent might have taken 40 steps, hallucinated three selectors, and clicked the same button eight times in a row.

The trajectory judge catches all of that:

CheckFails when
CoverageExpected tools missing from call sequence
HallucinationTool name not in the valid tool list
RedundancySame tool called 3+ times in a row
judge_trajectory.py
def judge(scenario_id, tool_calls, expected):
    sequence = [tc["name"]
                for tc in tool_calls]
    called = set(sequence)
    expected_set = set(expected)

    coverage = (
        len(expected_set & called)
        / len(expected_set)
    )
    hallucinated = [
        t for t in sequence
        if t not in VALID_TOOLS
    ]

    redundant = count_triple_repeats(sequence)

    return TrajectoryResult(
        coverage=coverage,
        hallucinated=hallucinated,
        redundant_calls=redundant,
        passed=(coverage >= 0.6
                and not hallucinated
                and redundant == 0)
    )
Putting It Together

The Eval Runner:
One Script, All Three Judges

eval_runner.py does three things:

1
Run the agent
Loads each golden scenario, runs the agent loop (MAX_TURNS = 8), captures the final finding text and every tool call made.
2
Apply all three judges
Deterministic + LLM + Trajectory — sequentially per scenario, results accumulated.
3
Write eval_report.md
Summary table + per-scenario rows + LLM judge detail. Optionally exports JSON for regression.

Measurement

Four Metrics
Worth Tracking

These four numbers give you a complete picture of agent health. Track them over time as you change models, prompts, or tools.

det pass rate
80%
Fraction of scenarios where keyword coverage ≥ threshold. Your primary correctness signal.
llm avg overall
0.74
Mean quality score across all scenarios. Drops when findings become vague or inaccurate.
traj pass rate
90%
Fraction of scenarios with valid tool sequences. A sudden drop = the model is hallucinating tool names.
hallucination rate
5%
Fraction of runs with at least one hallucinated tool call. Should stay close to zero.
Regression

Regression Harness:
Protect Your Baseline

The workflow:

1
Run evals → export metrics
After eval_runner.py, aggregate 4 key metrics into current.json.
2
Compare against baseline
regression.py loads baseline.json, checks each metric against its threshold, exits 1 on breach.
3
Promote when ready
python regression.py --save-baseline promotes current to the new floor. Commit it to the repo.

CI/CD

Evals in GitHub Actions:
eval.yml

Add a separate eval job that runs alongside your test agent:

  • Runs on every PR that touches 08-evaluations/ or agent code
  • Fails the build if regression.py exits 1
  • baseline.json is committed to the repo — it is your quality contract
  • Promoted only when the team deliberately decides to raise the bar
Cost note

10 scenarios × Haiku = ~$0.002 per eval run. Run the full suite on PRs, a subset (high severity only) on every push.

.github/workflows/eval.yml
name: Agent Evals
on:
  pull_request:
    paths:
      - '08-evaluations/**'
      - '**/agent.py'
      - '**/tools.py'

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          pip install anthropic
          python eval_runner.py
          python regression.py
        working-directory: 08-evaluations/python
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        if: always()
      - uses: actions/upload-artifact@v4
        with:
          name: eval-report
          path: 08-evaluations/python/eval_report.md
        if: always()

Key Takeaways

What You Learned
in Module 08

1
Three judges, three angles
Deterministic catches missing facts. LLM judge catches vague reasoning. Trajectory catches bad behavior. All three together give you confidence.
2
Golden dataset = your spec
expected_keywords and expected_tools are a machine-readable specification of what your agent must do. Treat them like unit tests — add to them when you find new failure modes.
3
baseline.json is a contract
Committing the baseline to the repo means every PR must maintain quality. Regressions fail the build before they reach production.
4
Evals cost almost nothing
10 scenarios with Haiku as judge = ~$0.002. There is no reason not to run evals on every PR. The cost of a production failure is not $0.002.
5
Each module has a checkpoint
Modules 01–07 each include a targeted eval checkpoint. Apply the right judge to the right module: trajectory for driver modules, LLM judge for the full framework.
6
Improve systematically
Evals turn gut-feel improvements into measured progress. Change a prompt, run evals, compare against baseline. Ship only if it's better.
