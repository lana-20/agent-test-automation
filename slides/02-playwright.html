Description quality matters

The model reads the description field to decide when to use a tool. Vague descriptions lead to wrong tool choices. Be specific about what the tool does and when.

tools.py · dispatch()

dispatch() routes
model calls to Playwright.

tools.py — dispatch()
def dispatch(name: str, inputs: dict, page: Page) -> str:
    if name == "navigate":
        page.goto(inputs["url"])
        return f"Navigated to {inputs['url']}"

    if name == "click":
        page.get_by_role(inputs["role"], name=inputs["text"]).click()
        return f"Clicked {inputs['role']} '{inputs['text']}'"

    if name == "fill":
        page.get_by_label(inputs["label"]).fill(inputs["value"])
        return f"Filled '{inputs['label']}' with '{inputs['value']}'"

    if name == "assert_visible":
        loc = page.get_by_role(inputs["role"], name=inputs["text"])
        try:
            loc.wait_for(state="visible", timeout=5000)
            return f"VISIBLE: {inputs['role']} '{inputs['text']}'"
        except PlaywrightTimeout:
            return f"FAIL: {inputs['role']} not visible"
Design Pattern

Return strings,
not exceptions.

❌ Don't do this

if name == "assert_visible":
    loc = page.get_by_role(...)
    loc.wait_for(state="visible")
    # Raises TimeoutError if not found
    # Agent loop crashes
    # No report, no context

✓ Do this

if name == "assert_visible":
    try:
        loc.wait_for(state="visible", timeout=5000)
        return "VISIBLE: button 'Add'"
    except PlaywrightTimeout:
        return "FAIL: button 'Add' not visible"
    # Model reads this string and decides
    # whether to report FAIL or retry
Key principle

Let the model decide how to handle failures — stop and report, or try a different approach. Don't make that decision in your Python code.

agent.py · System Prompt

Four rules that
shape the agent's behavior.

SYSTEM_PROMPT
SYSTEM_PROMPT = """
You are a senior test automation engineer.

Rules:
1. Work step by step. After each action,
   verify the result.
2. Prefer locating elements by ARIA role
   + visible text.
3. When an assertion fails, stop and
   report FAIL with the exact reason.
4. At the end, output a test report:
   overall PASS or FAIL on the first
   line, one bullet per step.
"""
Rule 1: Verify after each action
Without this, the model might chain actions without checking anything.
Rule 2: ARIA role + visible text
Reinforces the tool design — resilient locators that also test accessibility.
Rule 3: Surface failures immediately
Don't silently continue after a failed assertion.
Rule 4: Structured final report
Consistent format means result-parsing code can rely on it.
agent.py · The Loop

Twelve lines
that run everything.

agent.py
messages = [{ "role": "user", "content": task }]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=4096,
        system=SYSTEM_PROMPT, tools=TOOL_DEFINITIONS, messages=messages
    )
    messages.append({ "role": "assistant", "content": response.content })
    tool_results = []

    for block in response.content:
        if block.type == "text":
            print(f"\nAgent: {block.text}")
        elif block.type == "tool_use":
            result = dispatch(block.name, block.input, page)
            print(f"  -> {block.name}({block.input})\n     {result}")
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id, "content": result
            })

    if tool_results:
        messages.append({ "role": "user", "content": tool_results })
    if response.stop_reason == "end_turn":
        break

    name: string,

Extract types from libraries

Parameters<Fn>[N] extracts the Nth parameter type from any function. Stay in sync with the library without maintaining a manual union type.

Module 02 · Key Takeaways

Five things
to carry forward.

1
The loop is 12 meaningful lines
while True → call API → handle tool_use → check end_turn → break. That's the entire architecture.
2
Tools are your boundary
You define what the agent can do. The model decides how to use those tools. Keep descriptions specific.
3
Return strings from dispatch, not exceptions
Let the model decide what to do with failures — report, adapt, or retry.
4
ARIA role + visible text beats CSS selectors
Resilient to layout changes. Tests accessibility. The model already thinks in these terms.
5
The system prompt shapes behavior
"Work step by step, verify after each action, report FAIL and stop" — four words that determine how the agent handles uncertainty.
eval checkpoint

Eval Your Playwright Agent

Two things to measure after every run:

Tool accuracy

Did navigate / click / fill receive valid ARIA selectors? Track hallucinated selectors — any CSS class that doesn't exist on the page is a trajectory failure.

Finding correctness

Run on a page with a known bug. Score = keywords matched / keywords expected. Pass threshold: ≥ 60%.

# quick smoke eval from judge_deterministic import judge result = judge("TC-001", agent_finding, ["empty", "error", "password"]) print(result.score, result.missed) # 0.67 ['password']
