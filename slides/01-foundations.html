Core principle

The agent loop you write in Module 02 doesn't change in Module 06. One mental model. Eight applications.

The Problem

Scripted tests are
brittle by design.

What breaks

  • Hard-coded selectors — a renamed CSS class breaks the test
  • Positional locators — .todo-list li:nth-child(2) fails when order changes
  • No memory — a script doesn't know what happened two steps ago
  • Coverage gaps — writing scripted tests takes time; edge cases get skipped

What agents offer

  • Goal-oriented — describe what to verify, not how
  • ARIA-based locators — resilient to layout and styling changes
  • Full context — the messages array is the agent's memory
  • Exploratory coverage — the model reasons about what to check
Core Concept

A test agent reasons
its way through a task.

Instead of specifying every step, you give the agent a goal. The model decides what actions to take, verifies results, and adapts when something unexpected happens.

task description
-- Scripted test (you specify every step)
await page.locator('#todo-input').fill('Buy groceries')
await page.locator('.add-btn').click()
await page.locator('.todo-list li:nth-child(1)').click()

-- Agent task (you describe the goal)
"Go to TodoMVC, add 3 todos, mark one complete,
 verify the filter shows only that item."

The Loop

Twelve lines
of meaningful code.

pseudo-code
messages = [{ "role": "user", "content": task }]

while True:
    response = model.complete(messages, tools=TOOLS)

    # append what the model said
    messages.append({ "role": "assistant", "content": response.content })

    for block in response.content:
        if block.type == "tool_use":
            result = dispatch(block.name, block.input, page)
            tool_results.append({ "type": "tool_result", "content": result })

    if tool_results:
        messages.append({ "role": "user", "content": tool_results })

    if response.stop_reason == "end_turn":
        break

Core Vocabulary

A tool is a function
the agent can call.

You define the tools. The model decides when to call them and what arguments to pass, based on the description field.

  • name — what the model calls it
  • description — how the model decides when to use it
  • input_schema — validated JSON schema for the arguments
tool definition
{
  "name": "click",
  "description": "Click an element by
    ARIA role and visible text.",
  "input_schema": {
    "type": "object",
    "properties": {
      "role": { "type": "string" },
      "text": { "type": "string" }
    },
    "required": ["role", "text"]
  }
}
Core Vocabulary

Tool use and
stop reason.

tool_use
When the model wants to call a tool, the response includes a tool_use block with the tool name and chosen arguments.

Your code executes it and returns a result string.
block.type == "tool_use"
block.name   # → "click"
block.input  # → {"role":"button","text":"Add"}
stop_reason
Every API response includes a stop_reason. It tells you why the model stopped generating.
"tool_use" — execute tools, continue
"end_turn" — task complete, break the loop
Core Vocabulary

System prompt
and orchestrator.

System Prompt
A persistent instruction set that shapes how the agent behaves. Doesn't change during a run. Defines:
  • The agent's persona and role
  • Its rules and constraints
  • The expected output format
"You are a senior test automation engineer. Work step by step. When an assertion fails, stop and report FAIL with the exact reason."
Orchestrator
An agent that coordinates other agents. In Modules 05 and 06, the orchestrator:
  • Reads a feature description
  • Generates a structured test plan
  • Dispatches runner agents in parallel
The orchestrator plans. Runners execute. Separation of concerns.

Use both

Agents and scripted tests are complementary. Use scripted tests for stable, high-confidence critical paths. Use agents for coverage you couldn't afford to write by hand.

Module 01 · Key Takeaways

Five things
to carry forward.

1
The loop: observe → decide → act → repeat
Messages grow with every tool call. The model has full context. This pattern doesn't change in any module.
2
Tools are the boundary between model and browser
You define the tools. The model decides when and how to call them. Your dispatch function executes them.
3
stop_reason tells you when the agent is done
"tool_use" → keep going. "end_turn" → break the loop.
4
The system prompt shapes behavior, not the loop
Most tuning happens in the system prompt. The loop code stays the same.
5
Agents complement scripted tests — they don't replace them
Use agents for exploratory coverage and fast-changing surfaces. Keep scripted tests for stable critical paths.

Define "Good" Before You Build

Three questions every agent eval must answer — decide them now, not after your first production failure.

1
Correctness — Did it find the real bug? Keyword match against known expected findings.
2
Quality — Is the finding useful to a developer? LLM judge: accuracy + specificity + actionability.
3
Behavior — Did it use the right tools? No hallucinated selectors, no infinite loops.
You can't improve what you don't measure. Save baseline scores before you change the agent prompt or model.
