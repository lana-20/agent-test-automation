The Problem

Sequential is the
ceiling. Parallel is not.

With a single agent, scenarios run one at a time. 20 scenarios × 2 min each = 40 minutes.

With parallel runners, 20 scenarios take ~2 minutes — the time of the slowest one.

Architecture

One orchestrator.
Many runners.

  • Orchestrator reads a feature description, generates a structured test plan via tool_choice.
  • Runners execute scenarios. Each gets its own browser instance — fully isolated.
  • TestReport aggregates all results into a structured object.

models.py

Typed data models —
the API contract.

models.py
@dataclass
class TestScenario:
    id: str
    title: str
    steps: list[str]
    url: str

@dataclass
class RunnerResult:
    scenario_id: str
    title: str
    passed: bool
    summary: str
    step_log: list[dict]

@dataclass
class TestReport:
    scenarios: list[RunnerResult]
    # computed: .passed .failed .total

These dataclasses are the typed contract between the orchestrator and runners. When the orchestrator produces a TestScenario, the runner knows exactly what fields to expect.

Define models first

Shared data models prevent silent mismatches between what one agent produces and another expects. Rename a field? Python raises an error immediately.

orchestrator.py · Forced Structured Output

tool_choice enforces
guaranteed-valid JSON.

orchestrator.py — plan()
response = _CLIENT.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=2048,
    system=_SYSTEM,
    tools=[_PLAN_TOOL],
    tool_choice={ "type": "tool", "name": "create_test_plan" },
    # ↑ Forces the model to call this specific tool.
    #   Response ALWAYS contains a tool_use block.
    #   API validates output against the schema.
    messages=[{ "role": "user", "content": feature }],
)

tool_block = next(b for b in response.content if b.type == "tool_use")
raw = tool_block.input["scenarios"]
# ↑ Already a Python list matching the schema.
#   No JSON parsing. No regex. No "hope the model followed the format."

return [TestScenario(**s) for s in raw]
Use this pattern everywhere you need structured output

Don't prompt "output JSON in this format" — the model sometimes deviates. tool_choice enforces the schema at the API level. Zero parsing code.

run.py · Parallel Dispatch

Python: threads.
TypeScript: Promise.all.

run.py — ThreadPoolExecutor
from concurrent.futures import (
    ThreadPoolExecutor, as_completed
)

with ThreadPoolExecutor(
    max_workers=len(scenarios)
) as pool:
    futures = {
        pool.submit(run_scenario, s, headless): s
        for s in scenarios
    }
    for future in as_completed(futures):
        results.append(future.result())
Why threads, not asyncio?

Playwright's sync API is thread-safe but NOT asyncio-safe. Each thread runs its own sync_playwright() context. Don't use asyncio.gather().

index.ts — Promise.all
// TypeScript: naturally async
const results = await Promise.all(
    scenarios.map((s) => runScenario(s, headless))
);

// Each runScenario creates its own:
//   await chromium.launch({ headless })
//   await browser.newContext(...)
//   await context.newPage()
// No shared state between runs.

Module 05 · Key Takeaways

Five things
to carry forward.

1
tool_choice forces structured output
API validates against the schema. Guaranteed-valid JSON. Zero parsing code. Use this everywhere you need structured data from the model.
2
Shared data models are the API contract
Define TestScenario, RunnerResult, TestReport first. They constrain what the orchestrator produces and what runners consume.
3
Python: ThreadPoolExecutor. TypeScript: Promise.all
Don't use asyncio with sync Playwright. Each thread / async call needs its own browser instance.
4
Isolation is non-negotiable
No shared browser contexts between concurrent runners. Shared state causes flaky tests that are nearly impossible to debug.
5
[scenario_id] prefix in output is not optional
Without it, interleaved output from five runners is unreadable. Always prefix console output in parallel runs.
eval checkpoint

Eval Your Multi-Agent System

Parallelism introduces two new failure modes you must measure:

1
Inter-runner consistency — Run all 5 runners on the same scenario. Apply deterministic judge to each finding. If coverage varies by > 15%, the prompt is non-deterministic enough to be unreliable.
2
Throughput improvement — Compare wall-clock time: parallel run vs sequential simulation. Target ≥ 3× speedup for 5 runners. Less means contention or rate limiting.
3
tool_choice conformance — Verify forced structured output still parses cleanly under parallel load. Track JSON parse failures per 100 runs.
Consistency across runners is the multi-agent equivalent of test flakiness. Measure it before you add more runners.
